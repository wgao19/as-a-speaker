CSS Specifications
At this time you can no longer walk past speaking with me about CSS without hearing me advocating the CSS specifications. Please allow me to share a little bit of my personal story here.
Not long after I just started working as a front end developer, I worked on a design that contains a stamped edge. I didn‚Äôt know how to do it. So I did what I‚Äôve been doing since college ‚Äî I googled it. 

                .stamp:after {
                  content: '';
                  position: absolute;
                  left: 5px; top: 5px; right: 5px; bottom: 5px;
                  z-index: -1;
                }

I came across a number of solutions similar to [this one]. I thought I‚Äôd try to learn something out of it. After all, I‚Äôm a professional web developer now. I should not be the one just heartlessly copying and pasting snippets, right?

So as I tried to understand the code, I thought the line content: '' (content semicolon empty string) was irrelevant. Maybe the author forgot to take it off. So I left out that line and, expectedly, my stamp edge did not show. 


              background: red

I then went through a whole process of trying to find where that pseudo element went. I gave it a super big `width` and `height`, `background: red`, etc. I checked parent‚Äôs `position` because I recalled some relevancy there? But I simply could not find my pseudo element. I got so frustrated also because part of me did not put CSS as a top priority. My inner voice was like ‚ÄúPlease, just let me get over with it so I can work on ‚Äúmore interesting‚Äù stuff‚Äù.

My problem with this approach was that I did not understand where each line of code came from. And what can gravitate this issue with CSS is that more often than not *I thought I knew*.

When their computed content value is not none, these pseudo-elements generate boxes as if they were immediate children of their originating element, and can be styled exactly like any normal document-sourced element in the document tree.
A few years later I finally decided to read the CSS specifications, and it clearly stated that the pseudo-elements are generated when content value is not none.

Well anyway, looking back, I gained a lot of my imperative knowledge that way. Running into the problems, stack overflow, until I get very bored of following others' solutions. A lot of senior developers have even told me that our field has too many scattered pieces of knowledge that I have to master in order to do my job properly. Maybe, just maybe, it is not because the knowledge is scattered around by themselves. But how we find solutions to our problem broke them into scattered pieces?

But CSS specifications is one set of well-edited, systematic documentations on how browsers should render the page according to the stylesheets. You can hardly find any other literature that actually helps you build consistent and complete knowledge about this aspect of browser behaviors. The more you read the CSS specs, the more connection you'll be able to make with one another.

And furthermore, have you noticed that while some of us struggle to pass a so-called "plateau" and stop advancing from certain point of time, while some others seem to have kept advancing in their proficiency, and have you wondered why?

Now, I‚Äôd like to bring to your all's attention that we do not gain proficiency by following others‚Äô solutions. We gain proficiency by solving the problems by ourselves. And best guidance is one that specifies the features of your tools and lets you come up with the solutions by yourself. And CSS specifications are exactly this kind of guidance.

With this challenge series we put a lot of emphasis on the CSS specs. On alternate days we did not schedule any coding challenges at all, instead, we scheduled "reading time" of CSS specifications.
It is less difficult and more useful than it seems to read the CSS specs.
#specsreading

But in general, a personal lesson learned is this.

Each set of the CSS specifications may require some dedication for you to even begin to read it. But trust me, if have long-term goal as a front end developer, this is the most time-saving effort for you to master CSS. You'll gain cross-browser understanding about how CSS should be implemented from the vendors' and editors' perspective. And to share with you a secret, it brings this weird feeling of mastering CSS once you've read a few critical sections. And people will register you as someone who reads CSS specs, must be an expert.

And as a girl developer, sense of respect comes in rarer form than our guy peers. If we are to master something, do it like we mean it.
