How dynamic is your content?

With traditional website or blog site, you save everything in a database. Pages are generated by a server, and the server requests the data dynamically according to what the user requests.

With static site generators, everything is static and they get compiled into html pages at build time, and in case of Gatsby, they‚Äôre React components.

Would like to mention that in real life Gatsby sites compile contents not just from markdown files. There may be a mesh of content coming in from different sources including CMS and Gatsby handles them all.

Are the changes from content or behavior?

I guess one way to think about whether the dynamic part of your content comes from the changes of the content itself, or behavior of your users. Inventory stock is definitely dynamic with respect to user behavior, so it is with ‚Äúlikes‚Äù and comments.

Then what about blog posts, event or product descriptions. They won‚Äôt change unless you create something new. So when this is the case, maybe you don‚Äôt need a dynamic server after all. Is database necessarily better than a file to save your post?

I thought these are some questions that JAMStack urges us to think about.

Now, what if your product is a service or climbing gym multi passes that will never go out of stock as long as your are not out of business? Then all you need is a payment service fed in by an API and connected to your components, likely with hooks.
Do you have a ‚Äùdynamic‚Äù team of content generators?
If that‚Äôs the case, you probably need a CMS to manage content generation. Treating this as a separate process, we can feed our Gatsby sites with content from CMS. And if you look at the starters library you can see many starters are already powered by CMS.

Building web again?

I have an affection with static websites because that was how websites were born with.

Not many of us were from the web pages where HTML + CSS built the foundation of your website. Back then when you add new content to your website, you may need to write new HTML and CSS, to craft the webpage specifically for the new content. The code was naturally split by pages and directories, everything was server side rendered. Sometimes build dynamic servers for content but in general, the web technologies were quite straightforward.

Not all businesses act like or live in e-commerce platforms like Amazon, and not all the content we browse on the internet requires as complex features as to be called an application. There are solutions in all kinds of format and JAMStack just happens to be one of them that work, one that embraces edging technology in its own way and is accessible to more businesses.
